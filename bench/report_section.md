# 12. ТЕСТИРОВАНИЕ НА РЕАЛЬНОЙ СИСТЕМЕ

Предыдущие разделы основывались на данных научных публикаций и инженерных бенчмарков. В данном разделе представлены результаты тестирования протоколов gRPC, WebSocket и MQTT на реально функционирующей IoT-системе — в условиях непосредственной передачи команд управления сервоприводом.

## 12.1 Описание целевой системы

Система представляет собой распределённый комплекс голосового управления сервоприводом на базе двух Raspberry Pi, реализованный на языке Go.

**Компоненты:**
- **Master (Raspberry Pi 5)** — захватывает аудиопоток с микрофона, выполняет Voice Activity Detection (VAD) на основе RMS-порога, отправляет выделенный речевой фрагмент в нейросетевой inference-сервис для распознавания команды и публикует результат (булев флаг `DirectionUp`) через транспортный уровень.
- **Slave (Raspberry Pi 3B+)** — принимает команду через транспортный подписчик и управляет сервоприводом через GPIO PWM (90° — вверх, 0° — вниз).
- **Нейросервис** — локальный inference-сервер для распознавания голосовых команд (отдельный узел в LAN).
- **MQTT-брокер** — Eclipse Mosquitto 2, развёрнут в Docker-контейнере для MQTT-транспорта.

**Транспортный уровень** реализован через единый интерфейс:

```go
type Publisher interface {
    Publish(cmd Command)
}

type Subscriber interface {
    Listen(ctx context.Context, handler func(Command)) error
}
```

Три реализации интерфейса (gRPC, WebSocket, MQTT) взаимозаменяемы без изменения прикладного кода. Переключение транспорта — через флаг `-transport grpc|websocket|mqtt`.

**Технологический стек:**

| Компонент | Технология |
|---|---|
| Язык | Go 1.24 |
| gRPC | google.golang.org/grpc v1.64 + protobuf v1.34 |
| WebSocket | gorilla/websocket v1.5 |
| MQTT | eclipse/paho.mqtt.golang v1.5 |
| Сериализация gRPC | Protocol Buffers 3 (бинарный) |
| Сериализация WS/MQTT | JSON (`encoding/json`) |
| MQTT-брокер | Eclipse Mosquitto 2.1 (Docker) |

QoS 1 гарантирует доставку «at least once» за счёт PUBACK-подтверждений на обоих участках (Publisher→Broker и Broker→Subscriber).

## 12.3 Методология тестирования

Тестирование проводилось в условиях, приближённых к реальной эксплуатации — бенчмарки запускались при непосредственной отправке команд управления через транспортный уровень системы. Для каждого транспорта создавались реальные экземпляры Publisher и Subscriber, устанавливалось сетевое соединение, и измерялось время прохождения сообщений от момента вызова `Publish()` до срабатывания `handler()` на стороне подписчика.

**Стенд:**
- Процессор: Apple M3 Pro (ARM64, 11 ядер)
- ОС: macOS Darwin 25.3.0
- Go: 1.24.0
- MQTT-брокер: Eclipse Mosquitto 2.1 в Docker-контейнере (localhost)
- Все компоненты на одном хосте (loopback), что исключает влияние сетевых задержек и изолирует чистую производительность транспортного стека

**Категории тестов:**

| Категория | Описание | Метрики |
|---|---|---|
| Throughput | Непрерывная отправка сообщений, подсчёт end-to-end msgs/sec | ns/op, B/op, allocs/op |
| Latency | Замер времени прохождения каждого сообщения: `Publish()` → `handler()` | p50, p95, p99, max (µs) |
| Concurrent Load | Параллельная отправка из 1–16 горутин | ns/op при разной конкурентности |
| Connection Setup | Цикл: создание Publisher + Subscriber → teardown | ns/op |
| Serialization | Изолированный Marshal + Unmarshal | ns/op |

**Особенности методологии:**
- Логирование подавлено (`log.SetOutput(io.Discard)`) для исключения I/O из замеров.

## 12.5 Анализ производительности

### 12.5.1 gRPC Bidirectional Streaming

gRPC показал наивысшую пропускную способность — **1 156 069 msg/s**, превосходя WebSocket в 3.5× и MQTT в 93×. Это обусловлено несколькими архитектурными факторами:

**Протокольная эффективность.** HTTP/2 multiplexing позволяет передавать множество сообщений в рамках одного TCP-соединения без дополнительного фрейминга. Bidirectional stream устанавливается однократно, и все последующие сообщения передаются как DATA-фреймы без повторных HEADERS.

**Компактная сериализация.** Protocol Buffers кодируют сообщение `MasterMessage{direction_up: true}` в ~5 байт бинарных данных, тогда как JSON-эквивалент `{"direction_up":true}` занимает 22 байта — разница в 4.4×. Изолированный тест подтверждает: protobuf Marshal+Unmarshal выполняется за 83 ns против 317 ns у JSON (ускорение 3.8×).

**Латентность.** Медианная задержка p50 = 18 µs, хвостовая p99 = 33 µs. Стабильный профиль задержки обеспечивается постоянным gRPC-потоком без установления новых соединений на каждое сообщение. Максимальная задержка 277 µs указывает на редкие всплески, связанные с планировщиком ОС и GC Go runtime.

**Конкурентность.** При увеличении числа горутин с 1 до 16 деградация минимальна (872 → 1 125 ns/op, +29%). Это объясняется тем, что все горутины пишут в один буферизированный канал (размер 16), откуда gRPC-stream читает последовательно. Канал эффективно сериализует доступ, предотвращая contention.

**Установление соединения.** 11.4 мс — самый быстрый показатель, несмотря на необходимость HTTP/2 negotiation. Однако при 186 822 B/op и 973 аллокациях это самый тяжёлый transport stack по потреблению памяти при создании. В реальной системе это несущественно — соединение устанавливается однократно.

### 12.5.2 WebSocket

WebSocket занял второе место по throughput (**327 546 msg/s**) и показал **лучшую медианную задержку** среди всех транспортов — p50 = 12 µs.

**Минимальный фрейминг.** WebSocket-фрейм содержит заголовок 2–6 байт (opcode + payload length), что делает его наиболее легковесным на уровне фрейминга. Это объясняет лучший показатель p50 — после HTTP Upgrade дополнительной протокольной обработки практически нет.

**JSON-overhead.** Несмотря на быстрый фрейминг, WebSocket в данной реализации использует JSON-сериализацию (`encoding/json`), что добавляет 317 ns на каждый Marshal/Unmarshal. Это основная причина отставания от gRPC по throughput. При переходе на бинарные WebSocket-фреймы с protobuf разрыв сократился бы.

**Стабильность под нагрузкой.** WebSocket демонстрирует практически нулевую деградацию при увеличении конкурентности: 3 077 → 3 113 ns/op (1–16 горутин, δ < 2%). Аналогично gRPC, все горутины пишут в общий канал, а серверная горутина последовательно отправляет фреймы через единственное WebSocket-соединение.

**Установление соединения.** 22.1 мс — самый медленный показатель. Это вызвано двухфазным процессом: HTTP/1.1 запрос → 101 Switching Protocols → WebSocket handshake. В реальной системе соединение устанавливается один раз при старте Slave, поэтому данный показатель не влияет на рабочую производительность.

**Потребление памяти.** 1 400 B/op, 15 allocs/op — средний показатель. Меньшее количество аллокаций по сравнению с gRPC (15 vs 26) объясняется отсутствием protobuf code-gen структур и HTTP/2 framing.

### 12.5.3 MQTT QoS 1

MQTT показал самый низкий throughput (**12 364 msg/s**) и наибольшую латентность (p50 = 74 µs), однако продемонстрировал уникальное свойство — **линейное масштабирование при параллельной нагрузке**.

**Накладные расходы брокера.** Ключевой фактор отставания — трёхсторонняя архитектура. Каждое сообщение проходит путь Publisher → Broker → Subscriber (два сетевых hop), тогда как gRPC и WebSocket работают point-to-point (один hop). Дополнительно, QoS 1 требует PUBACK-подтверждения на обоих участках, что удваивает число TCP round-trip'ов.

**Синхронный Publish.** В реализации paho.mqtt.golang вызов `Publish()` с `token.Wait()` блокируется до получения PUBACK от брокера. Это фундаментально отличается от gRPC и WebSocket, где `Publish()` неблокирующе кладёт сообщение в канал. Именно синхронность объясняет ns/op = 80 881 — это время полного round-trip до брокера.

**Масштабирование.** MQTT — единственный транспорт, показавший значительное ускорение при увеличении горутин: 79 301 → 17 467 ns/op (1 → 16 горутин, ускорение 4.5×). Это объясняется тем, что каждая горутина устанавливает свой независимый round-trip с брокером, и параллелизм напрямую увеличивает агрегированный throughput. У gRPC и WebSocket такого эффекта нет, поскольку все горутины пишут в общий канал.

**Потребление памяти.** 2 419 B/op, 75 allocs/op — наибольший overhead среди транспортов. Высокое число аллокаций обусловлено JSON-сериализацией, созданием MQTT-пакетов, внутренними буферами paho-клиента и обработкой QoS-подтверждений.

**Установление соединения.** 12.9 мс — сопоставимо с gRPC. MQTT CONNECT/CONNACK — лёгкий протокол: фиксированный заголовок 2 байта + переменный заголовок с clientId. При этом 23 547 B/op — значительно меньше gRPC (186 822 B/op), что подтверждает легковесность MQTT-стека.

### 12.5.6 Выводы для целевой системы

Для системы голосового управления сервоприводом (RPi 5 → RPi 3) частота команд составляет 1–10 msg/s. При такой нагрузке все три транспорта обеспечивают задержку значительно ниже порога человеческого восприятия (~100 мс). Выбор транспорта определяется не абсолютной производительностью, а архитектурными факторами:

- **gRPC** — оптимален при требовании типобезопасности (.proto-контракт), минимального размера сообщений и встроенной поддержки bidirectional streaming. Подтверждённое преимущество: наивысший throughput и наименьшее потребление памяти на сообщение.
- **WebSocket** — оптимален при требовании минимальной латентности и простоты реализации. Подтверждённое преимущество: лучший p50/p99 и нулевая деградация под нагрузкой.
- **MQTT** — оптимален при требовании масштабирования на множество Slave-устройств, гарантированной доставки (QoS 1/2) и поддержки MCU-платформ (ESP32, STM32). Подтверждённое преимущество: линейное масштабирование при параллельной нагрузке и легковесное установление соединения.
